### Post Copy

Built and shipped `StudioFlow`, a multi-location SaaS platform for service-based businesses like salons, tattoo studios, barbershops, and wellness teams.

The goal was to build something that feels like a real product, not just a dashboard UI. StudioFlow combines an internal operations workspace with a customer-facing booking experience, so teams can manage appointments, clients, payments, consent forms, reminders, analytics, notifications, and audit history in one system.

What’s included:
- JWT auth with role-based and studio-scoped access control
- multi-location support and admin onboarding
- internal scheduling, payments, forms, notifications, analytics, and audit logs
- public booking flow by studio and location
- customer self-service reschedule and cancel flow
- deployment-ready config, staging support, seeded demo data, and smoke-test coverage

Tech stack:
React, TypeScript, Tailwind CSS, Framer Motion, React Router, Java 17, Spring Boot, Spring Security, JWT, JPA

This was a strong exercise in balancing product design, operational realism, and practical architecture choices without overengineering.

## Technical Architecture Post

### Suggested Hook

One of the most interesting parts of building `StudioFlow` was making it behave like a real multi-user SaaS product.

### Post Copy

One of the most interesting parts of building `StudioFlow` was treating it like a real multi-user SaaS product instead of just a collection of CRUD screens.

Some of the engineering decisions I focused on:
- JWT auth with backend-enforced role and studio scoping
- a separate public booking flow instead of exposing internal routes to customers
- multi-location support without introducing heavy organizational complexity
- audit logs and activity history for accountability
- notification and reminder foundations with staged email/SMS delivery support
- environment-based local, staging, and production configuration
- realistic demo seed data and smoke-test coverage for critical backend flows

The stack is React + TypeScript on the frontend and Spring Boot + Spring Security + JPA on the backend.

I wanted the project to show both product thinking and systems thinking: not just “can it render,” but “does it behave like software a team could actually operate with.”
